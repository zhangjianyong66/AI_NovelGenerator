# 支持批量草稿定稿审校

## 目标

让生成任务页的章节范围参数不只服务于“批量定稿”，而是支持三个明确的批量章节操作：批量草稿、批量定稿、批量审校。用户配置起始章节和结束章节后，可以按章节范围推进真实创作流程，不需要逐章手动切换当前章节号。

## 用户价值

- 用户想一次性生成第 2-10 章草稿时，系统应直接创建对应 `chapter_X.txt`，而不是要求逐章修改“当前章节号”。
- 用户已有多章草稿后，可以继续批量定稿，并看到每章成功或失败的汇总。
- 用户想检查多章一致性时，可以批量审校，并在任务日志中保留逐章审校结果。

## 已确认事实

- 生成任务页当前有单章按钮 `草稿`、`定稿`、`审校`，以及一个笼统的 `批量` 按钮。证据：`frontend/src/features/generation/components/GenerationActions.vue:35`。
- 生成任务页当前的批量参数固定按批量定稿校验，范围内缺少章节文件时会阻止提交。证据：`frontend/src/pages/GenerationPage.vue:196`。
- 前端生成任务类型当前只有 `batch`，没有区分批量草稿、批量定稿、批量审校。证据：`frontend/src/services/types.ts:43`。
- API 当前将 `batch` 标题定义为“批量定稿章节”，并在 API 层要求范围内章节文件都存在。证据：`app/api/server.py:251`、`app/api/server.py:1152`。
- 执行器当前的 `batch` 只调用 `_run_batch_finalization(...)`，逐章复用 `_run_finalization(...)`。证据：`app/services/generation_executor.py:66`、`app/services/generation_executor.py:274`。
- 单章草稿执行器不要求根部章节文件预先存在，成功后会把 legacy `chapters/chapter_X.txt` 同步为根部 `chapter_X.txt`。证据：`app/services/generation_executor.py:182`、`app/services/generation_executor.py:206`。
- 单章审校执行器读取章节正文并把审校结果写入任务日志，不修改项目文件。证据：`app/services/generation_executor.py:336`。
- 现有项目文档明确“批量第一版只定稿范围内已有 `chapter_X.txt`”，本任务需要更新该合同。证据：`docs/feature-map-and-acceptance.md:62`。

## 需求

- R1：保留单章 `草稿`、`定稿`、`审校` 按钮，继续使用当前章节号。
- R2：新增或改造批量入口，至少提供 `批量草稿`、`批量定稿`、`批量审校` 三个明确动作，避免“批量”语义不清。
- R3：三种批量操作共用起始章节、结束章节、目标字数、最低字数、自动扩写参数；其中自动扩写只对定稿阶段有实际作用。
- R4：批量草稿应允许目标范围内缺少章节文件，并在成功后创建/同步根部 `chapter_X.txt`。
- R5：批量草稿遇到范围内已存在的 `chapter_X.txt` 时默认跳过，不覆盖正文，并在日志中记录跳过章节。
- R6：批量定稿和批量审校应要求目标范围内章节文件已存在且可读；缺失时应在前端或后端给出中文可操作错误。
- R7：批量执行时单章失败后继续后续章节，最终任务状态按“全部成功为 done；任一失败为 failed”返回，并在日志中汇总成功/失败章节；批量草稿的“跳过已有章节”不算失败。
- R8：前端在真实后端不可写时禁用批量写操作，不新增 mock 写入 fallback。
- R9：任务历史继续写入本地 SQLite 状态库，记录原始请求参数、日志、错误和状态。
- R10：更新项目文档和 AGENTS.md 中会被后续复用的批量生成能力说明。

## 非目标

- 不把生成任务改成后台异步 worker。
- 不新增失败任务重试或“只重跑失败章节”。
- 不新增章节生命周期状态数据库，例如“草稿/已定稿/已审校”持久状态。
- 不改变旧 GUI 入口 `python main.py` 的行为。
- 不迁移章节正文到 SQLite；`chapter_X.txt` 仍是正文权威来源。

## 验收标准

- [ ] 生成任务页显示单章 `草稿`、`定稿`、`审校` 和批量 `批量草稿`、`批量定稿`、`批量审校`。
- [ ] 只存在 `chapter_1.txt` 时，选择范围 `2-10` 创建批量草稿不会因缺少 `chapter_2.txt` 失败；有效配置下会逐章生成草稿文件。
- [ ] 批量草稿范围内已有 `chapter_X.txt` 时不覆盖已有正文，任务日志记录跳过章节，并继续生成缺失章节。
- [ ] 只存在 `chapter_1.txt` 时，选择范围 `2-10` 创建批量定稿或批量审校会提示缺少章节文件。
- [ ] 批量草稿、批量定稿、批量审校都在任务日志中记录章节范围、逐章结果、成功/失败汇总。
- [ ] 批量审校成功时不修改章节正文、摘要、角色状态或剧情要点文件。
- [ ] 后端 API 测试覆盖三种批量阶段的成功、部分失败和关键前置条件错误。
- [ ] 前端类型检查和构建通过。
