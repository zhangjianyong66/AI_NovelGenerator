## Context

项目是 Python CustomTkinter 桌面 GUI，主入口为 `main.py`，主窗口由 `ui/main_window.py` 初始化，各标签页和区域分散在 `ui/` 下的多个模块中构建。当前界面大量控件直接传入 `font=("Microsoft YaHei", 12)`，主要章节编辑区使用 14 号字体。截图显示在高分辨率窗口下，标签页、右侧配置区、按钮和输入项整体偏小，影响阅读和点击。

本次改动采用用户确认的“舒适版”：提升可读性，但不做视觉重设计，不改变页面结构和业务流程。

## Goals / Non-Goals

**Goals:**

- 通过统一字体和控件缩放提升整个 GUI 的可读性。
- 将普通标签、按钮、输入框、下拉框等控件提升到舒适阅读尺寸。
- 将长文本编辑、日志和预览类区域设置为略大的字体，便于写作和校对。
- 尽量集中样式定义，减少后续维护硬编码字号的成本。
- 保持当前标签页、左右布局、生成流程和配置保存逻辑不变。

**Non-Goals:**

- 不重做 UI 视觉风格、配色、控件层级或导航结构。
- 不新增用户可配置的主题/字号设置。
- 不修改 LLM、Embedding、WebDAV、小说生成、文件读写等业务逻辑。
- 不调整配置文件 schema。

## Decisions

### 使用集中式 UI 样式模块

新增或使用一个轻量样式模块集中定义字体族、普通字号、文本区字号、小号辅助字号和缩放比例。各 UI 构建模块优先引用这些常量，而不是继续散落 `("Microsoft YaHei", 12)`。

替代方案是直接批量把 12 改为 15。该方案实现快，但会继续保留分散硬编码，后续二次调整成本高。因此采用集中式常量。

### 在应用启动时设置 CustomTkinter 缩放

在创建主窗口后、构建界面前设置 `ctk.set_widget_scaling(1.15)`。这会让按钮、输入框、标签页等控件尺寸和间距随字体提升同步变大，避免只放大文字导致控件拥挤。

替代方案是只改字体，不改控件缩放。该方案风险更低，但右侧配置区按钮和输入框可能出现视觉拥挤，不符合“舒适版”目标。

### 字号采用“舒适版”分层

普通控件使用 15 号左右字体，主要长文本编辑区使用 16 号左右字体，小型帮助按钮和次要说明使用 12-13 号字体。这样既提升整体可读性，又避免所有控件等比例变大后信息密度下降过多。

### 保持窗口结构，必要时扩大右侧区域

当前主窗口为左右两栏，右侧为配置和参数区域。字号放大后，如果右侧输入项明显挤压，应适度增加右侧配置区域的最小宽度或相关控件宽度；左侧主要编辑区继续占据剩余空间。

## Risks / Trade-offs

- 字体放大后右侧区域可用宽度不足 → 通过适度增加右侧最小宽度、保持滚动区域、检查主要页面来缓解。
- 不同操作系统缺少 `Microsoft YaHei` → 保持现有字体族策略，不在本次引入跨平台字体探测；CustomTkinter/Tk 会按系统回退处理。
- 大量 UI 文件仍可能存在少量硬编码字体遗漏 → 通过 `rg "Microsoft YaHei"` 检查并优先覆盖主要界面、对话框和角色库入口。
- 自动化测试难以完全验证视觉效果 → 结合 `python -m pytest tests`、应用启动检查和截图人工核验。
