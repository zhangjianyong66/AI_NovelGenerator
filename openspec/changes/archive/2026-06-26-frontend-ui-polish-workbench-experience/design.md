## Context

此前前端变更已经建立 Tauri 2 + Vue 3 + TypeScript 工作台骨架，并完成真实本地 API 接入。当前 UI 的问题不在“有没有功能”，而在“功能如何组织”：页面组件粒度偏大，表单和状态反馈重复，用户从项目准备到写作、生成、校对和资料管理的路径还不够清楚。

本变更的第一阶段应专注页面结构和组件复用，不扩大业务能力边界。

## Target Information Architecture

前端主导航仍保留项目、工作台、章节、生成、知识库和设置，但它们的职责需要更明确：

```text
项目
  └─ 选择/确认输出目录和当前项目入口
      ↓
工作台
  └─ 汇总当前写作焦点、关键文件、章节焦点和生成状态
      ↓
章节
  └─ 章节列表、章节正文编辑、字数和保存状态
      ↓
生成
  └─ 创建任务、配置批量参数、查看任务列表和日志
      ↓
知识库
  ├─ 知识文件
  ├─ 剧情要点
  └─ 角色库
      ↓
设置
  ├─ 项目参数
  ├─ LLM
  ├─ Embedding
  ├─ 代理与阶段模型
  └─ WebDAV
```

工作台页不是所有功能的复制入口，而是当前项目状态和下一步动作的聚合页。复杂配置留在设置页，复杂资料管理留在知识库页，复杂任务操作留在生成页。

## Component Strategy

优先抽取“有重复行为和一致状态需求”的组件，而不是为了视觉拆分机械创建组件。

候选组件：

- `PageHeader`：页面标题、副标题、右侧主操作区。
- `ActionBar`：一组主要/次要操作按钮，统一禁用和 loading 呈现。
- `StatusMessage`：成功、错误、警告、加载和空状态提示。
- `FormSection`：带标题、说明、操作区的配置分组。
- `ConfirmPanel` 或 `ConfirmAction`：清理向量库、WebDAV 恢复等显式确认操作。
- `FileList` / `ChapterList`：文件、章节、角色等列表选择模式。
- `LongTextEditor`：长文本编辑、只读查看、字数统计和保存状态。
- `Tabs`：设置页和知识库页的分组切换。

组件应保持无业务后端知识：页面负责调用 store/service，组件负责输入输出、布局和状态展示。

## Page Direction

### Settings

设置页从单页高密度表单调整为分组或 tabs。第一阶段建议 tabs：

- 项目参数：输出路径、主题、类型、章节数、当前章节、章节上下文参数。
- LLM：配置列表、当前配置详情、新增/删除、测试。
- Embedding：配置详情和切换向量库提醒。
- 生成阶段：阶段到 LLM 配置映射，可与代理同页或独立 tab。
- WebDAV：连接信息、测试、备份、恢复。

保存策略可以先保留“保存全部”，但 UI 应明确哪些分组有可保存内容。后续可再拆为分组保存。

### Knowledge

知识库页使用 tabs：

- 知识文件：导入资料、清理向量库、导入结果和文件列表。
- 剧情要点：`plot_arcs.txt` 的只读长文本视图、字数和缺失状态。
- 角色库：分类列表、角色选择、角色正文编辑、导入角色、写入章节参数。

这能避免角色库编辑器把知识导入和剧情查看挤到同一视觉层级。

### Generation

生成页组织为稳定操作面板：

```text
┌────────────────────────────────────────────┐
│ 创建任务：设定 目录 草稿 定稿 审校 批量      │
├────────────────────────────────────────────┤
│ 批量参数/当前选择：章节范围 字数 自动扩写    │
├──────────────────────┬─────────────────────┤
│ 任务列表             │ 选中任务日志/错误    │
└──────────────────────┴─────────────────────┘
```

任务列表不应把所有日志完全铺开；更稳定的结构是列表负责扫描，详情面板负责日志、错误和结果引用。

## State Feedback

统一状态模式：

- 加载中：按钮和区域状态都应可见，避免只禁用按钮。
- 成功：靠近触发操作展示短消息，不长期占据主内容。
- 错误：显示用户可理解原因，保留页面可继续操作。
- 空状态：用于无章节、无任务、无剧情要点、无角色时的页面区域。
- 确认：破坏性或覆盖性操作必须显式确认，避免直接依赖 `window.confirm` 作为长期方案。

## Risks / Trade-offs

- 过早抽象组件可能让页面开发变慢：只抽取跨 2 个以上页面或带一致状态行为的组件。
- tabs 过多可能隐藏重要配置：页面 header 或 tab label 应暴露保存/错误状态。
- 不改业务契约会限制某些 UX 优化：第一阶段接受限制，避免前后端同时变动造成验证困难。
- 生成任务当前仍是最小后端任务模型：UI 可以稳定结构，但不假设已有实时日志或取消能力。

## Verification

- 运行 `cd frontend && npm run typecheck`。
- 运行 `cd frontend && npm run build`。
- 启动前端和本地 API 后检查项目、工作台、章节、生成、知识库、设置页面可打开。
- 检查后端不可用时 mock/disconnected 状态仍清楚可见。
- 检查设置、知识库和生成页在桌面宽度与窄屏布局下无明显文本重叠。
